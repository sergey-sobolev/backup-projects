# Резервное копирование проектов (rsync + YAML)

Утилита `backup-projects` копирует заданные каталоги на локальный носитель или удалённый хост через **rsync**, используя конфигурацию в **YAML**. Поддерживаются режимы инкремента, отдельной копии с меткой времени и архивов `.tgz`. После успешного прогона создаётся **флаг успеха**; все шаги пишутся в **журнал** (файл и/или stderr).

Проект распространяется под лицензией [MIT](LICENSE).

## Требования

- **ОС:** Linux
- **Python:** 3.9+
- **Внешние программы:** `rsync`, `tar` (для режима `tgz`)
- **Python-пакеты:** `PyYAML` (см. `requirements.txt`)

Для расписания по умолчанию используется **systemd** (пользовательские unit’ы); альтернатива — **cron** (скрипт выводит строку для `crontab`).

## Установка

### Вариант A: подготовка системы (Debian/Ubuntu) и виртуальное окружение

```bash
sudo make system-deps    # rsync, python3-yaml, python3-venv, python3-pip
make deps                # создаёт .venv и ставит пакет в режиме разработки + pytest
```

Установка только зависимостей пакетного менеджера (без venv):

```bash
sudo apt-get install -y rsync python3-yaml python3-pip
pip install --user -r requirements.txt
```

### Вариант B: установка из исходников (`setup.py`)

```bash
pip install .
# или в режиме разработки
pip install -e ".[dev]"
```

После установки в `$PATH` появится команда `backup-projects`.

### Запуск из клонированного репозитория без установки

```bash
./backup-projects -c /path/to/backup-config.yaml
```

Скрипт в корне добавляет каталог репозитория в `PYTHONPATH` и вызывает тот же код, что и установленная команда.

## Настройка

1. Скопируйте пример конфигурации:

   ```bash
   cp config.example.yaml backup-config.yaml
   ```

2. Отредактируйте `backup-config.yaml`:

   - `target` — **цель по умолчанию** (локальный путь или `user@host:/path`); используется, если у источника не заданы свои цели
   - `default_mode` — `update` | `copy` | `tgz` для источников без своего `mode` (для совместимости читается и устаревший ключ `mode`, если `default_mode` не задан)
   - `sources` — список каталогов: строка или объект с `path`, опционально **`enable`** (`false` — пропустить источник), `name` и **`mode`**. Цели: **`target`** или **`targets`**. Элемент `targets` — строка пути или объект **`{ target: путь, mode?, rotate?, max_count? }`**. Если заданы и `target`, и `targets`, учитывается только **`targets`**. Для режима **`tgz`**: **`rotate: true`** и **`max_count`** (целое ≥ 1) — после успешной выкладки архива на **локальный** каталог оставить не больше `max_count` новейших по времени файлов `name-*.tgz` / `name_*.tgz` и удалить остальные; приоритет настроек: объект в `targets` → источник → корень YAML. Для **удалённой** цели `user@host:...` ротация **не выполняется** (в журнал пишется предупреждение)
   - `success_flag` — относительный путь файла-флага от **корневого** `target` (общий флаг после всех копий, не привязан к per-source целям)
   - `log_filename` — имя файла журнала в каталоге состояния (`$XDG_STATE_HOME/backup-projects/` или `~/.local/state/backup-projects/`); по умолчанию `backup.log`
   - `log_file` — полный путь к журналу (перекрывает только имя из `log_filename`), либо `true` / `false` / отсутствие ключа — см. раздел «Журнал»
   - опционально: `sync_delete`, `rsync_extra`, `tgz_datetime_suffix` (корень) — для `tgz`: при `true` имя архива `name_YYYYMMDDHHMMSS.tgz`; иначе `name-YYYYmmdd_HHMMSS.tgz`
   - опционально: **`rotate`**, **`max_count`** — см. описание `sources` (только локальные цели для `tgz`)
   - опционально: **`max_workers`** — число потоков для параллельного запуска **задач по целям** (каждая пара источник→цель — отдельная задача). По умолчанию `min(8, число_задач)`. Значение обрезается сверху числом задач

## Параллельность

Все задачи резервного копирования (каждая цель для каждого источника) выполняются в **пуле потоков** (`ThreadPoolExecutor`). Логирование потокобезопасно. Флаг **`success_flag`** выставляется только после **успешного** завершения **всех** задач. При ошибке в одной из задач остальные в этом запуске всё равно дорабатывают до конца контекста пула, затем ошибка пробрасывается наверх.

## Примеры использования

Инкрементальное обновление на флэшку:

```bash
backup-projects -c ~/backup-config.yaml
```

Указать файл журнала и подробный вывод в консоль:

```bash
backup-projects -c ./backup-config.yaml -l /var/log/backup-projects.log -v
```

Тихий stderr (уровень WARNING), журнал — из конфига или по умолчанию в файл:

```bash
backup-projects -c ./backup-config.yaml -q
```

Запуск как модуля (удобно в CI или при `PYTHONPATH`):

```bash
python3 -m backup_projects -c backup-config.yaml
```

Расписание (systemd для пользователя):

```bash
./install-backup-schedule.sh
```

Только строка для cron:

```bash
./install-backup-schedule.sh --cron
```

## Удалённый сервер: SSH и rsync

Копирование на другой хост делается **тем же rsync**: в `target`, `target` у источника или в `targets` указывается строка вида **`пользователь@хост:/путь`** (абсолютный путь на удалённой стороне). Rsync для таких целей по умолчанию запускает **`ssh`** — это тот же защищённый канал, что используют **scp** и **sftp** в типичной установке OpenSSH; отдельные вызовы `scp`/`sftp` в проекте не нужны.

Готовый пример YAML: **[config.remote.example.yaml](config.remote.example.yaml)**.

### Что подготовить на машине, откуда запускается бэкап

1. **SSH-доступ** без интерактива (для cron/systemd): пара ключей, при необходимости `ssh-agent`, либо в `rsync_extra` передать `-e 'ssh -o BatchMode=yes …'`.
2. Проверка: `ssh пользователь@хост` и `rsync -av /etc/hostname пользователь@хост:/tmp/` (или тестовый каталог).
3. На сервере каталог назначения должен **существовать** (или родитель с правом создания), права на запись — у указанного пользователя.
4. При смене хоста/ключей — обновление `~/.ssh/known_hosts`, если включена проверка ключей.

### Нестандартный порт и опции ssh

Через **`rsync_extra`** (строка или список), например:

```yaml
rsync_extra: ["-e", "ssh -p 2222 -o BatchMode=yes"]
```

Алиас хоста из **`~/.ssh/config`** (`Host mynas` → `target: mynas:/var/backups/...`) тоже подходит: rsync передаст имя в `ssh`.

### Флаг успеха на удалённой стороне

Если корневой **`target`** удалённый, файл **`success_flag`** создаётся **на том же хосте и в том же логическом каталоге**, что и `target` (через rsync загрузки маленького файла).

### Ротация архивов (tgz) и удалённые цели

**`rotate` / `max_count`** действуют только если каталог назначения архива — **локальный путь**. Для `user@host:/path` старые `.tgz` на сервере скрипт **не удаляет** (нужен был бы отдельный доступ по SSH).

## Журнал (log)

- В файл пишутся сообщения уровня **DEBUG** (команды rsync/tar и шаги).
- В **stderr** по умолчанию — **INFO** и выше; `-v` включает DEBUG; `-q` оставляет WARNING и ошибки.
- **Имя файла** в общем конфиге: `log_filename` (только basename, путь к каталогу фиксирован: `…/backup-projects/<log_filename>`).
- **Полный путь** к журналу: строка в `log_file`; она имеет приоритет над `log_filename`.
- `log_file: false` — не писать в файл (только stderr). `log_file: true` или отсутствие / `null` — файл в каталоге состояния с именем из `log_filename` (по умолчанию `backup.log`).
- Опция **`-l` / `--log-file`** переопределяет и `log_file`, и `log_filename`.
